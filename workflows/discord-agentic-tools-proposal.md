# Discord 自然語言工具化（Agentic Function Calling）提案

> 狀態：**提案（尚未實作）** ｜ 建立日期：2026-06-05
> 目的：讓 Discord Bot 能處理「依條件找人/找訊息再執行管理動作」的自然語言指令，
> 例如「把這 2 天內講過髒話的人禁言 30 秒」，而非只能對**明確 @ 指定的單一對象**操作。

---

## 1. 背景與動機

目前 Discord Bot 採 function calling：使用者輸入 → AI 回傳要呼叫的工具 + JSON 參數 → 後端 `DiscordToolService.ExecuteAsync` 執行。現有 23 個工具（禁言、踢除、清訊息、查成員/頻道/身分組、稽核日誌…）皆設計成**明確單一對象**（參數帶 `<@使用者ID>`）。

實測痛點：使用者必須 @標註對象才能執行；無法表達「把符合某條件的一群人」這種需求。

## 2. 現況限制（為什麼目前做不到）

| 限制 | 說明 |
|---|---|
| 單輪執行 | `CompleteWithToolsAsync` 為單發：AI 決定一次 → 後端執行 → 結束。**不會把工具結果餵回 AI 再決策**，無法「先抓人再執行」。 |
| 無讀訊息工具 | 查詢類工具只有成員/頻道/身分組/稽核日誌，**沒有「讀取/掃描頻道訊息內容」**。 |
| 無訊息資料源 | Bot 走 Slash Command `/chat`，只收到「對 Bot 說的話」；一般頻道聊天不會進後端，`ExternalMessages` 也只存與 Bot 的對話。 |
| 動作工具單一對象 | 如 `timeout_member` 參數 `target` 為單一 `<@id>`，不能吃「一組算出來的人」。 |

## 3. 願景：Agentic、依「必要參數是否足夠」自動決策

核心觀念：**函式呼叫本來就會看 `required` 參數夠不夠**。差別在「湊不齊時怎麼辦」——

- 目前單輪：湊不齊只能反問使用者或放棄。
- 目標多步：湊不齊時，AI **自己用其他工具去解析/補齊**（先抓訊息、找出對象），再執行動作。

→ 不需要寫死 if/else 判斷情境；AI 依參數是否足夠自動選擇「直接做」或「先解析再做」。

| 使用者輸入 | AI 行為 | 步數 |
|---|---|---|
| 「禁言 @bob 30 秒」 | `target` 已明確 → 直接 `timeout_member(@bob, 30s)` | 1 步 |
| 「把 2 天內講髒話的人禁言 30 秒」 | `target` 缺 → `fetch_recent_messages(48h)` → 讀內容判斷違規者 → 對每人 `timeout_member(…, 30s)` | 多步 |

此模式**通用化**：任何「依條件找人/找訊息再執行」都適用（例：「把今天沒發言的人移到 AFK」「把提到特價的訊息釘選」），不限髒話情境。

## 4. 核心架構改動：單輪 → 多步工具迴圈（ReAct）

唯一必須補的關鍵：把 `IAiService.CompleteWithToolsAsync` 從「單發」改為「迴圈」：

```
送出（系統提示 + 使用者訊息 + 工具定義）
  → 收到 tool calls → 後端執行 → 把結果以 tool/role 訊息接回對話
  → 再送出 → …重複，直到 AI 回純文字（無 tool call）或達最大步數
```

## 5. 髒話/不當內容判斷：由 AI 原生完成

只要提供「讀訊息工具」，AI 在迴圈中拿到訊息內容後**自己判斷**哪些違規——AI 的理解力即分類器。
- 優點：免維護關鍵字清單、可判模糊情況（人身攻擊、引戰等）、貼合「讓 AI 判斷」需求。
- 對照：也可改走「關鍵字清單」或「獨立分類 AI 呼叫」，但通用性與彈性較低（見 §8 路線比較）。

## 6. 需新增/修改（規劃層級，尚未實作）

1. **多步工具迴圈（核心）**：`CompleteWithToolsAsync` 改為迴圈；保留單輪相容或以參數控制最大步數。
2. **讀取工具** `fetch_recent_messages(channel?, since_hours, limit)` → 回 `[{authorId, name, content, ts}]`；預設用呼叫當下頻道 `ctx.ChannelId`。
3. **迴圈護欄**：最大步數、最大掃描則數、最多影響人數。
4. **批次破壞性動作確認**：多人 timeout 前走「確認按鈕」，並以**伺服器端暫存待確認名單**（TTL，`custom_id` 帶 token）避免按確認時重算/AI 重複花費。
5. 既有 `TimeoutMemberAsync` 等動作工具直接重用（逐一呼叫）。

## 7. 取捨與護欄

- **成本/延遲**：多步 = 多次 AI 呼叫、且要把訊息內容餵回（token 變多）→ 必設則數/長度上限。
- **15 分鐘窗口**：Discord 互動已採 deferred（type:5「思考中…」）+ 背景處理 + followup PATCH；迴圈與批次須控制在 15 分鐘內完成。
- **誤判**：AI 可能誤判 → 確認步驟為最後防線，回報附「依據的訊息片段」供管理員核對。
- **權限階級**：Bot 無法禁言階級更高者/管理員/伺服器擁有者 → 跳過並列入失敗回報。
- **排除對象**：Bot 自己、其他 Bot；是否排除指令發起人待定。
- **Rate limit**：批次動作逐一送、加小延遲，逐筆回報成功/失敗。

## 8. 路線比較

| | 多步 agent 迴圈（建議） | 各別複合工具 |
|---|---|---|
| 通用性 | 高，任何「找→做」自動適用 | 低，每情境各寫一個工具 |
| AI 判斷 | 原生（讀訊息自己判） | 需工具內另呼叫 AI 分類 |
| 架構改動 | 較大（要做迴圈） | 較小（沿用單輪） |
| 成本可預測性 | 較低，需嚴格上限 | 較高 |

**建議**：走多步 agent 迴圈，最貼合「AI 自己判斷情況、參數夠就直接做、不夠先抓」的提案，且一勞永逸適用未來各種功能。

## 9. Discord 平台前置（非程式）

- 開發者後台開啟 **MESSAGE CONTENT 特權 intent**（否則 REST 抓回的訊息 `content` 為空）。
- Bot 權限需含 **Read Message History** + **Timeout Members**（依用到的動作再加對應權限）。

## 10. 待拍板事項

1. 確定走**多步 agent 迴圈**這條路？（大方向決策）
2. 護欄初值：最大步數、最多掃描則數、最多影響人數各抓多少？
3. 髒話判斷用 AI 原生（建議）還是關鍵字清單？
4. 掃描範圍預設「呼叫當下頻道」，是否要開放全伺服器（成本/rate limit 較高）？
5. 是否排除指令發起人與管理員（預設排除）？
6. 分類/判斷用該 Bot 設定的模型即可，還是固定用較強模型？
