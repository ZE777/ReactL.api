# 後端工程規則（資深後端工程師觀點）

## 非同步與效能

### N+1 查詢問題

```csharp
// ❌ 迴圈內查詢 → N+1
foreach (var conv in conversations)
{
    var messages = await _context.Messages
        .Where(m => m.ConversationId == conv.Id)
        .ToListAsync();
}

// ✅ 一次載入關聯資料
var conversations = await _context.Conversations
    .Include(c => c.Messages)
    .Where(c => c.BotId == botId)
    .ToListAsync();
```

### 只查需要的欄位

```csharp
// ❌ 撈整個 Entity 卻只用部分欄位
var personas = await _context.Personas.ToListAsync();
return personas.Select(p => p.Name);

// ✅ 用 Select 投影
var names = await _context.Personas
    .Select(p => new { p.Id, p.Name })
    .ToListAsync();
```

### 分頁查詢必須加 Skip/Take

```csharp
// ✅ 分頁標準寫法
var items = await _context.Conversations
    .OrderByDescending(c => c.CreatedAt)
    .Skip((page - 1) * pageSize)
    .Take(pageSize)
    .ToListAsync();
```

## 資料驗證

### 在 Service 層做業務驗證，Controller 做格式驗證

```csharp
// Controller：只處理格式（DataAnnotations 或 FluentValidation）
public async Task<IActionResult> Create([FromBody] CreatePersonaRequest req)
{
    if (!ModelState.IsValid)
        return BadRequest(ApiResponse<object>.Fail("請求格式錯誤"));
    // ...
}

// Service：處理業務規則
public async Task<PersonaResponse> CreateAsync(CreatePersonaRequest request)
{
    if (request.Name.Length > 100)
        throw new ValidationException("名稱不得超過 100 字");
    if (await _repo.ExistsByNameAsync(request.Name))
        throw new ConflictException("名稱已存在");
}
```

## 安全規則

### 禁止在 Log 中記錄敏感資料

```csharp
// ❌
_logger.LogInformation("使用者登入：{Email} 密碼：{Password}", email, password);

// ✅
_logger.LogInformation("使用者登入：{Email}", email);
```

### 資料庫查詢禁止字串拼接（防 SQL Injection）

```csharp
// ❌ 永遠不要這樣寫（即使在 EF Raw SQL 中）
var sql = $"SELECT * FROM Personas WHERE Name = '{name}'";

// ✅ 參數化查詢
var personas = await _context.Personas
    .Where(p => p.Name == name)
    .ToListAsync();

// ✅ 必要時使用 FromSqlInterpolated（自動參數化）
var result = await _context.Personas
    .FromSqlInterpolated($"SELECT * FROM Personas WHERE Name = {name}")
    .ToListAsync();
```

### API 輸入永遠不信任

```csharp
// ❌ 直接使用用戶傳入的 ID 查詢並回傳
public async Task<IActionResult> Get(int id) =>
    Ok(await _context.Personas.FindAsync(id));

// ✅ 驗證資源是否屬於當前用戶（多租戶或有權限邏輯時）
public async Task<IActionResult> Get(int id)
{
    var userId = User.GetUserId();
    var result = await _service.GetByIdForUserAsync(id, userId);
    if (result == null) return NotFound(...);
    return Ok(...);
}
```

## 資料庫操作規則

### UpdatedAt 由程式控制，不依賴資料庫 Trigger

```csharp
// ✅ 在 SaveChanges 前統一更新
public override Task<int> SaveChangesAsync(CancellationToken ct = default)
{
    foreach (var entry in ChangeTracker.Entries<BaseEntity>())
    {
        if (entry.State == EntityState.Modified)
            entry.Entity.UpdatedAt = DateTime.UtcNow;
    }
    return base.SaveChangesAsync(ct);
}
```

### 刪除優先考慮軟刪除

```csharp
// 針對有稽核需求的資料（對話紀錄、用戶資料）使用軟刪除
public abstract class BaseEntity
{
    public int Id { get; set; }
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
    public DateTime UpdatedAt { get; set; } = DateTime.UtcNow;
    public bool IsDeleted { get; set; } = false;  // 軟刪除旗標
}
```

## 架構初步審查（資深後端工程師觀點）

以下是對當前後端規劃的審查，標記影響可靠性、效能、安全性的遺漏：

| 項目 | 狀態 | 說明 |
|------|------|------|
| 自訂例外類別 | ❌ 缺少 | 需要建立 `Common/Exceptions/` 資料夾，定義 `AppException`（基底）、`NotFoundException`（404）、`ConflictException`（409）、`ValidationException`（400），讓 Middleware 根據型別對應 HTTP 狀態碼 |
| HttpClient 外部 API 呼叫 | ❌ 缺少 | 呼叫 AI API 需要透過 `IHttpClientFactory` 注入，並設定 Timeout / Retry Policy（Polly），避免服務崩潰影響整體 API |
| 健康檢查端點 | ❌ 缺少 | 需要 `/health` 端點，確認 DB 連線狀態，部署時自動檢測可用性 |
| Rate Limiting | ❌ 缺少 | AI 類端點每次呼叫費用高，必須加 Rate Limiting 防止濫用（.NET 8 內建 `RateLimiter`） |
| 快取策略 | ⚠️ 未規劃 | `Personas` 等讀多寫少的資料建議加 `IMemoryCache`，降低 DB 負擔 |
| 全域查詢過濾器 | ⚠️ 未規劃 | 若使用軟刪除，需要在 `OnModelCreating` 加 `HasQueryFilter(e => !e.IsDeleted)`，否則每個查詢都要手動加 Where 條件 |
| CORS 設定 | ❌ 缺少 | 前台（Next.js）呼叫後台 API 時會被瀏覽器阻擋，需在 `Program.cs` 設定允許的 Origin |

## 回應格式規則

- 永遠回傳 `ApiResponse<T>`，不直接回傳裸資料
- 刪除成功回傳 `204 No Content`，不需要包 `ApiResponse`
- List 查詢沒有資料時回傳 `200 + []`，不回傳 `404`
- 單筆查詢找不到時回傳 `404`
