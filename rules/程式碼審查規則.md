# 程式碼審查規則（Code Reviewer 觀點）

## 送審前自我檢查清單

提交 PR 前，開發者自行確認以下項目：

- [ ] 命名是否符合 `naming-conventions.md`
- [ ] Controller 沒有商業邏輯
- [ ] Service 沒有直接使用 DbContext
- [ ] 沒有殘留的 `Console.WriteLine` 或 `Debug.WriteLine`
- [ ] 沒有被註解掉的廢棄程式碼
- [ ] 所有公開方法都有對應的 Log
- [ ] 沒有把 API Key、密碼、Token 硬寫在程式碼裡

## Controller 層審查重點

```csharp
// ❌ 商業邏輯不應該在 Controller 裡
[HttpGet("{id}")]
public async Task<IActionResult> Get(int id)
{
    var persona = await _context.Personas.FindAsync(id);
    if (persona == null) return NotFound();
    persona.LastAccessedAt = DateTime.UtcNow;   // ← 這行屬於 Service 層
    return Ok(persona);
}

// ✅ Controller 只負責接收、委派、回傳
[HttpGet("{id}")]
public async Task<IActionResult> Get(int id)
{
    var result = await _service.GetByIdAsync(id);
    if (result == null) return NotFound(ApiResponse<object>.Fail("Persona 不存在"));
    return Ok(ApiResponse<PersonaResponse>.Ok(result));
}
```

## Service 層審查重點

```csharp
// ❌ Service 不應直接操作 DbContext
public class PersonaService : IPersonaService
{
    private readonly AppDbContext _context;   // ← 違反依賴規則

// ✅ Service 只依賴 IRepository 介面
public class PersonaService : IPersonaService
{
    private readonly IPersonaRepository _repo;
    private readonly ILogger<PersonaService> _logger;
```

## 錯誤處理審查

```csharp
// ❌ Controller/Service 不需要到處包 try-catch
public async Task<IActionResult> Create([FromBody] CreatePersonaRequest req)
{
    try { ... }
    catch (Exception ex) { return StatusCode(500, ex.Message); }  // ← 由 Middleware 統一處理
}

// ✅ 只有在「預期內的業務錯誤」才需要明確處理
public async Task<PersonaResponse> CreateAsync(CreatePersonaRequest request)
{
    if (await _repo.ExistsByNameAsync(request.Name))
        throw new InvalidOperationException("Persona 名稱已存在");
    // 其他未預期例外讓 ExceptionMiddleware 接管
}
```

## 非同步使用審查

```csharp
// ❌ 常見錯誤：async void、.Result、.Wait()
public async void Save() { ... }           // 例外無法被捕捉
var result = _repo.GetAllAsync().Result;   // 可能造成 deadlock

// ✅ 正確用法
public async Task SaveAsync() { ... }
var result = await _repo.GetAllAsync();
```

## 架構初步審查（Code Reviewer 觀點）

以下是對當前後端規劃的審查，標記出可能影響日後 PR 品質的遺漏：

| 項目 | 狀態 | 說明 |
|------|------|------|
| 自訂例外類別 | ❌ 缺少 | 缺少 `NotFoundException`、`ConflictException`、`ValidationException`，Service 層目前只能 throw 通用 `Exception`，審查時無法辨別錯誤類型 |
| Request DTO 驗證 | ❌ 缺少 | DTO 上沒有 `[Required]`、`[MaxLength]` 等 DataAnnotations，或 FluentValidation，Controller 收到的資料沒有格式保護 |
| 常數集中管理 | ❌ 缺少 | `Common/` 資料夾規劃中，但未定義 `AppConstants.cs`，目前會散落 Magic String（如預設 AI Model 名稱） |
| 測試專案 | ❌ 缺少 | 無 Unit Test / Integration Test 專案，送審時無法驗證 Service 邏輯的正確性 |
| 分頁 DTO | ❌ 缺少 | 列表 API 沒有 `PagedRequest` / `PagedResponse<T>`，大量資料時會全量回傳 |

## 常見退件原因

| 問題 | 說明 |
|------|------|
| 方法過長（>30 行） | 應拆分為更小的私有方法 |
| 多個職責在同一個方法 | 違反 Single Responsibility |
| 魔術數字 / 魔術字串 | 提取為常數或設定檔 |
| 回傳 `null` 而非空集合 | List 方法應回傳 `[]` 不回傳 `null` |
| DTO 直接回傳 EF Entity | 必須轉換為 Response DTO |
| 未處理 `null` 輸入 | 介面邊界需驗證 |
