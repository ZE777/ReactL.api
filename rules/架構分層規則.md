## 層與層的依賴方向

```
HTTP 請求
    ↓
Request DTO（驗證輸入格式與欄位限制）
    ↓
Controller（呼叫 Service）
    ↓ 依賴 IXxxService 介面
Service（商業邏輯，直接注入 AppDbContext）
    ↓
Entity（SQL 資料表完整對應）
    ↓
Service 將 Entity 投影為 Domain 物件（過濾敏感欄位、加入 JOIN 欄位）
    ↓
Controller 將 Domain 物件 map 為 Response DTO
    ↓
ApiResponse<T> 包裝後回傳前端
```

- Controller 只能依賴 `IXxxService` 介面，不直接使用 DbContext
- Service 直接注入 `AppDbContext`（無獨立 Repository 層）
- Service 的回傳型別為 **Domain 物件**，不回傳 Entity 也不回傳 Response DTO
- Controller 負責將 Domain map 成 Response DTO，Service 不知道 DTO 的存在
- Monitor 和 AI Service 例外：產出聚合統計或 SSE 協議物件，直接回傳 DTO

## 各層職責邊界

| 層 | 可以做 | 不可以做 |
|----|--------|----------|
| Controller | 接收請求、驗證 ModelState、呼叫 Service、Domain → Response DTO、回傳 ApiResponse | 寫商業邏輯、直接操作 DbContext |
| Service | 商業邏輯、查詢 DbContext、Entity → Domain 投影、呼叫外部 AI API | 處理 HTTP、回傳 Response DTO |
| Domain | 代表業務物件（純屬性類別） | 含 DB 實作細節、含敏感欄位（PasswordHash、加密 Token） |
| Request DTO | 定義輸入格式、DataAnnotations 驗證 | 含商業邏輯、含回應欄位 |
| Response DTO | 定義 API 合約、欄位選擇與命名 | 含商業邏輯、直接等同 Entity |
| Middleware | 攔截請求/回應、統一錯誤格式 | 寫商業邏輯 |

## 三層物件的職責對照

| 物件類型 | 位置 | 特性 |
|---------|------|------|
| Entity | `Models/` | 完整對應 SQL 欄位，含 PasswordHash、BotTokenEncrypted 等敏感欄位 |
| Domain | `Domain/` | 業務物件，Service 回傳型別，已過濾敏感欄位，可帶 JOIN 計算欄位 |
| Request DTO | `DTOs/Requests/` | 前端輸入，有 `[Required]`、`[MaxLength]` 驗證 |
| Response DTO | `DTOs/Responses/` | API 回傳合約，按情境選取欄位（列表 / 詳情分開） |
