# 系統架構設計規則（資深系統架構師觀點）

## 架構原則

### 依賴方向不可逆轉

```
HTTP 請求
    ↓
Request DTO（輸入驗證）
    ↓
Controller（路由、Domain → Response DTO 轉換）
    ↓ 依賴介面（IXxxService）
Service（商業邏輯、Entity → Domain 投影）
    ↓ 直接注入 AppDbContext（無 Repository 層）
DbContext（EF Core）
    ↓
SQL Server
```

任何層都不能反向依賴上層，也不能跨層直接呼叫。  
Service 的回傳型別是 **Domain 物件**，Controller 收到 Domain 後自行 map 成 Response DTO。

### 介面先行（Interface First）

新增功能時，永遠先定義介面再實作：
- `Services/Personas/IPersonaService.cs` → 再寫 `Services/Personas/PersonaService.cs`

介面定義回傳型別時，使用 Domain 物件（例如 `Task<PersonaDomain>`），不使用 Entity 也不使用 Response DTO。

## 關注點分離（Separation of Concerns）

| 關注點 | 負責的地方 |
|--------|-----------|
| 路由對應 | Controller |
| 輸入驗證格式 | Request DTO（DataAnnotations） |
| 回應格式定義 | Response DTO |
| 業務物件（無敏感欄位） | Domain |
| 商業規則 | Service |
| 資料查詢 / 持久化 | Service 直接使用 AppDbContext |
| 跨切面邏輯（Log、錯誤、認證） | Middleware |
| 資料表結構 | EF Entity + Migration |

**禁止「神物件（God Object）」**：一個類別承擔多個關注點。

## 外部服務整合規則

呼叫外部 API（AI、Email、第三方）一律封裝成 Service + Interface：

```
IAiService  →  AiService（呼叫 OpenAI / Claude API）
IEmailService → EmailService（呼叫 SendGrid）
```

不在 Controller 或其他 Service 直接呼叫 `HttpClient`，集中管理重試、逾時、錯誤處理。

## 設定管理規則

| 設定類型 | 存放位置 |
|----------|----------|
| 非敏感預設值 | `appsettings.json` |
| 本機開發設定 | `appsettings.Development.json` |
| 敏感資訊（API Key、DB 密碼） | `.NET User Secrets`（開發）/ 環境變數（Production） |

**原則：敏感資訊不能進版控，不能寫死在程式碼裡。**

## 資料庫設計原則

### 每張表都需要的基礎欄位（BaseEntity）

```
Id          uniqueidentifier PRIMARY KEY（應用層 Guid.NewGuid() 產生，非資料庫自增）
CreatedAt   DATETIME2 NOT NULL（UTC，SaveChangesAsync 自動填入，新增後不覆寫）
UpdatedAt   DATETIME2 NOT NULL（UTC，新增與修改時都更新）←  AuditableEntity 才有
IsDeleted   BIT NOT NULL DEFAULT 0  ← 僅 SoftDeletableEntity（可軟刪除資料表）才加
DeletedAt   DATETIME2 NULL          ← 搭配 IsDeleted
```

三層基底類別：`BaseEntity`（Id + CreatedAt）→ `AuditableEntity`（+ UpdatedAt）→ `SoftDeletableEntity`（+ IsDeleted + DeletedAt）

### 外鍵與關聯設計

- 每個外鍵都要有對應的導航屬性
- 集合導航屬性初始化為空集合（`= []`），避免 null 判斷
- 多對多關係建立明確的中間資料表 Entity（例如 `BotPersonaBinding`）

### 索引建立時機

在以下情境建立索引：
- WHERE 條件常用欄位
- 外鍵欄位（EF 自動處理）
- 排序欄位（`CreatedAt`、`UpdatedAt`）
- 唯一性約束欄位（`Email`、`Slug`）

## API 版本策略

路由統一使用 `/api/v1/` 前綴，為未來版本升級預留空間。  
當需要破壞性變更時，新增 `/api/v2/` 而非修改現有端點。

## 中介層（Middleware）職責清單

| Middleware | 職責 |
|-----------|------|
| `ExceptionMiddleware` | 捕捉所有未處理例外，統一格式化為 500 回應並寫 Log |
| `RequestLoggingMiddleware` | 記錄每個請求的 Method、Path、StatusCode、耗時 |
| JWT Authentication | 驗證 Bearer Token，注入 Claims |
| CORS | 允許前端來源跨域請求 |

Middleware 執行順序（`Program.cs` 中由上到下）：
```
Exception → CORS → Authentication → Authorization → Routing → Endpoints
```

## 橫切關注點（Cross-Cutting Concerns）

以下屬於橫切關注點，**不應分散在各個 Controller/Service 裡**：

- **日誌**：由 Serilog + RequestLoggingMiddleware 統一處理
- **錯誤處理**：由 ExceptionMiddleware 統一處理
- **認證/授權**：由 JWT Middleware 統一處理
- **輸入驗證格式**：由 ModelState 統一處理
- **資料庫 Timestamp 更新**：由 DbContext.SaveChangesAsync 統一處理

## 可維護性原則

- 每個方法不超過 30 行；超過則提取私有方法
- 單一職責：一個 Service 方法只做一件事
- 避免深層巢狀（if 層數超過 3 層應重構）
- Magic Number / Magic String 提取為常數：`public static class AppConstants`
