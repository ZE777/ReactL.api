# 系統架構設計規則（資深系統架構師觀點）

## 架構原則

### 依賴方向不可逆轉

```
HTTP 請求
    ↓
Controller（輸入驗證、路由）
    ↓ 依賴介面（IXxxService）
Service（商業邏輯）
    ↓ 依賴介面（IXxxRepository）
Repository（資料存取）
    ↓
DbContext（EF Core）
    ↓
SQL Server
```

任何層都不能反向依賴上層，也不能跨層直接呼叫。

### 介面先行（Interface First）

新增功能時，永遠先定義介面再實作：
- `Interfaces/IXxxRepository.cs` → 再寫 `Repositories/XxxRepository.cs`
- `Interfaces/IXxxService.cs` → 再寫 `Services/XxxService.cs`

好處：方便日後替換實作（例如換快取策略、換 ORM），不影響上層。

## 關注點分離（Separation of Concerns）

| 關注點 | 負責的地方 |
|--------|-----------|
| 路由對應 | Controller |
| 請求/回應格式 | DTO（Request / Response） |
| 商業規則 | Service |
| 資料查詢 / 持久化 | Repository |
| 跨切面邏輯（Log、錯誤、認證） | Middleware |
| 資料表結構 | EF Entity + Migration |

**禁止「神物件（God Object）」**：一個類別承擔多個關注點。

## 外部服務整合規則

呼叫外部 API（AI、Email、第三方）一律封裝成 Service + Interface：

```
IAiService  →  AiService（呼叫 OpenAI / Claude API）
IEmailService → EmailService（呼叫 SendGrid）
```

不在 Controller 或其他 Service 直接呼叫 `HttpClient`，集中管理重試、逾時、錯誤處理。

## 設定管理規則

| 設定類型 | 存放位置 |
|----------|----------|
| 非敏感預設值 | `appsettings.json` |
| 本機開發設定 | `appsettings.Development.json` |
| 敏感資訊（API Key、DB 密碼） | `.NET User Secrets`（開發）/ 環境變數（Production） |

**原則：敏感資訊不能進版控，不能寫死在程式碼裡。**

## 資料庫設計原則

### 每張表都需要的基礎欄位（BaseEntity）

```
Id          INT IDENTITY PRIMARY KEY
CreatedAt   DATETIME2 NOT NULL DEFAULT GETUTCDATE()
UpdatedAt   DATETIME2 NOT NULL DEFAULT GETUTCDATE()
IsDeleted   BIT NOT NULL DEFAULT 0    ← 需要稽核的資料表才加
```

### 外鍵與關聯設計

- 每個外鍵都要有對應的導航屬性
- 集合導航屬性初始化為空集合（`= []`），避免 null 判斷
- 多對多關係建立明確的中間資料表 Entity（例如 `BotPersonaBinding`）

### 索引建立時機

在以下情境建立索引：
- WHERE 條件常用欄位
- 外鍵欄位（EF 自動處理）
- 排序欄位（`CreatedAt`、`UpdatedAt`）
- 唯一性約束欄位（`Email`、`Slug`）

## API 版本策略

路由統一使用 `/api/v1/` 前綴，為未來版本升級預留空間。  
當需要破壞性變更時，新增 `/api/v2/` 而非修改現有端點。

## 中介層（Middleware）職責清單

| Middleware | 職責 |
|-----------|------|
| `ExceptionMiddleware` | 捕捉所有未處理例外，統一格式化為 500 回應並寫 Log |
| `RequestLoggingMiddleware` | 記錄每個請求的 Method、Path、StatusCode、耗時 |
| JWT Authentication | 驗證 Bearer Token，注入 Claims |
| CORS | 允許前端來源跨域請求 |

Middleware 執行順序（`Program.cs` 中由上到下）：
```
Exception → CORS → Authentication → Authorization → Routing → Endpoints
```

## 橫切關注點（Cross-Cutting Concerns）

以下屬於橫切關注點，**不應分散在各個 Controller/Service 裡**：

- **日誌**：由 Serilog + RequestLoggingMiddleware 統一處理
- **錯誤處理**：由 ExceptionMiddleware 統一處理
- **認證/授權**：由 JWT Middleware 統一處理
- **輸入驗證格式**：由 ModelState 統一處理
- **資料庫 Timestamp 更新**：由 DbContext.SaveChangesAsync 統一處理

## 架構初步審查（資深系統架構師觀點）

### 現有規劃完整度評估

**已規劃** ✅
- 分層架構（Controller → IService → IRepository → DbContext）
- 介面先行（Interfaces/ 資料夾）
- 統一回應格式（ApiResponse<T>）
- 統一例外處理（ExceptionMiddleware 規劃中）
- 日誌策略（Serilog + 每日輪換）
- SQL 腳本版本管理（SqlScripts/Init + Changes）
- DI 生命週期規範

**遺漏或待補強** ❌

| 遺漏項目 | 優先級 | 影響 |
|---------|--------|------|
| 自訂例外類別層次 | 🔴 高 | ExceptionMiddleware 無法根據例外型別對應正確 HTTP 狀態碼 |
| CORS 政策設定 | 🔴 高 | 前端完全無法呼叫 API |
| Request DTO 驗證 | 🔴 高 | 無格式保護，任何格式的資料都能進入 Service 層 |
| JWT 認證設定 | 🔴 高 | 「規劃中」但無細節，所有 API 目前都是公開的 |
| Swagger / OpenAPI 設定 | 🟡 中 | 沒有 Swagger 就沒有可視化的 API 文件與測試介面 |
| 健康檢查（Health Checks） | 🟡 中 | 部署環境無法自動偵測服務是否正常 |
| AI API 整合模式 | 🟡 中 | `IAiService` / `AiService` 尚未定義呼叫方式、Timeout、重試策略 |
| 分頁機制 | 🟡 中 | `PagedRequest` / `PagedResponse<T>` 通用 DTO 缺少 |
| 全域查詢過濾器 | 🟢 低 | 軟刪除若沒有過濾器，每個查詢都要手動加條件 |
| 環境區分設定（appsettings） | 🟢 低 | `appsettings.Development.json` 範本已規劃，但需要補上所有設定的鍵名 |

### 建議補充順序

```
1. 自訂例外類別（AppException 體系）
2. Request DTO DataAnnotations
3. CORS 設定
4. Swagger 設定
5. JWT 設定
6. Health Checks
7. 分頁 DTO
8. IAiService 定義 + HttpClient 設定
```

## 可維護性原則

- 每個方法不超過 30 行；超過則提取私有方法
- 單一職責：一個 Service 方法只做一件事
- 避免深層巢狀（if 層數超過 3 層應重構）
- Magic Number / Magic String 提取為常數：`public static class AppConstants`
