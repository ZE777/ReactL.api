# 版本與發布規範（Versioning & Release）

> 適用範圍：ReactL Prompt Studio 全端（後端 `ReactL.api`、前端 `ReactL`：admin + web）。
> 前後端**版本號同步**——同一個產品版本代表同一批可一起運作的前後端。

---

## 1. 版本號格式：語意化版本 `vX.Y.Z`（SemVer）

| 段 | 名稱 | 代表意思 | 何時 +1 |
|---|---|---|---|
| **X** | 主版本（MAJOR） | 重大里程碑 / **不相容變更**（破壞性 API、需手動資料遷移、架構大改） | 破壞相容性時；同時 Y、Z 歸 0 |
| **Y** | 次版本（MINOR） | **新增功能**（向下相容） | 加新功能時；同時 Z 歸 0 |
| **Z** | 修訂（PATCH） | **修錯 / 小調整**（不新增功能：bug fix、文案、設定微調） | 只修錯/微調時 |

- **起始版本：`v1.0.0`** —— 視為第一個功能完整的正式定版（後台 / 前台 / Bot / BYOK / 存取碼 / Discord agent 工具 / 內建角色權限 / 模型推薦皆齊備）。
- 規則速記：新功能 → `Y+1`、修錯 → `Z+1`、破壞性 → `X+1`。

## 2. 前後端同步

- 前端（admin + web）與後端**共用同一個版本號**，代表「這一版產品」。
- 同一次發布：有改動的一方標該版本號；兩邊都有改動則都標同號。
- 版本號同時寫入：
  - 後端：`ReactL.api/ReactL.api.csproj` 的 `<Version>`
  - 前端：`prompt-studio-admin/package.json`、`prompt-studio-web/package.json` 的 `version`

## 3. Commit 訊息規範

每個「定版」commit 的**首行開頭寫版本號**，接既有的 Conventional Commits 類型：

```
v1.0.0 feat(discord): 自然語言 agent 工具（掃訊息→找人→批次禁言）
v1.0.1 fix(discord): 修正多選名單並行 race
v1.1.0 feat(admin): 模型推薦提示
```

- 格式：`vX.Y.Z <type>(<scope>): <說明>`
- `type`：`feat` / `fix` / `docs` / `refactor` / `style` / `chore` …
- 一個版本可含多個檔案/子項，首行寫主軸，細節寫 body。

## 4. 發布流程

1. 確認功能完成、編譯/測試通過。
2. 依變更性質決定新版本號（§1）。
3. 更新版本檔（csproj / package.json）。
4. **同步更新對應的規劃/功能文件**至實作現況。
5. Commit（首行帶版本號）→ push 到對應的 GitHub。
6.（可選）打 git tag `vX.Y.Z`。
